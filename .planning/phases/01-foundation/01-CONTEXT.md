<decisions>
## Implementation Decisions

### Config structure
- YAML config file with two top-level sections: `site` (metadata, base URL, politeness settings) and `levels` (array of level definitions with depth, selectors, pagination rules, field mappings)
- Config structure follows the architecture research pattern: site block defines global settings, levels array defines per-depth navigation and extraction rules
- Pydantic models validate config on load -- each section has its own model (SiteConfig, LevelConfig, FieldMapping) composed into a root ScrapeConfig model
- Validation errors are field-level and actionable: "Level 2 missing required field 'link_selector'" not "ValidationError: 1 error"
- Config is immutable after loading -- components receive it at init and never mutate it
- Example configs live in a `configs/` directory with an annotated example showing all options

### CLI design
- Single command, no subcommands -- this is a single-purpose batch tool, not a framework
- Invocation: `scraper <config-path> [--overrides]` using Typer with the config file path as a required positional argument
- CLI overrides for one-off runs: `--output-dir`, `--delay-min`, `--delay-max`, `--max-pages`, `--log-level` (debug/info/warning), `--dry-run`
- `--help` auto-generated by Typer from type hints
- CLI args override the corresponding config values -- config file provides defaults, CLI provides overrides
- Exit codes: 0 = success, 1 = config validation error, 2 = runtime error

### Logging and terminal output
- structlog for all logging: JSON format to log file, pretty colored output to terminal
- Log file written to output directory as `scrape.log` with JSON lines (one event per line)
- Default log level: INFO. `--log-level debug` for verbose, `--log-level warning` for quiet
- Every log event includes: timestamp, event name, and relevant context (URL, level name, error type when applicable)
- Terminal progress display during crawl: line-by-line status updates showing current level, pages processed, error count
- Progress display is minimal in Phase 1 -- full rich/tqdm progress bars can be added as polish later. Phase 1 focuses on the structured logging infrastructure that all subsequent phases depend on

### Dry-run mode
- Triggered by `--dry-run` flag
- Validates config fully (schema + field types + selector syntax where possible)
- Fetches one page per configured level to verify selectors work against the actual site
- Prints extracted fields in a readable table format to terminal (field name, selector used, value found or "NOT FOUND")
- Does NOT write output files or log files
- Does NOT follow pagination or links to next level
- Exits with code 0 if all levels extract at least one field, code 1 if config/selector issues found

### robots.txt compliance
- Fetch and parse robots.txt from the target site's base URL on startup using urllib.robotparser
- Check every URL against robots.txt rules before requesting -- disallowed URLs are skipped and logged
- Respect Crawl-delay directive if present in robots.txt (overrides config delay if robots.txt delay is longer)
- Cache robots.txt for the entire run -- fetch once, reuse
- If robots.txt fetch fails (404, timeout), log a warning and proceed (assume allowed) -- this matches standard scraper behavior

### Request delays
- Randomized delay between min and max values (default 1.0s to 3.0s) configured in YAML under `site.request_delay.min` and `site.request_delay.max`
- Delay applied between every request (not per page batch)
- CLI overrides: `--delay-min` and `--delay-max`
- Use random.uniform for true randomization within the range
- If robots.txt Crawl-delay is present and longer than config delay, use the robots.txt value as the minimum

### Claude's Discretion
- Exact structlog processor chain configuration
- Terminal color scheme and formatting choices
- Internal module organization within the recommended project structure
- Test fixtures and mocking strategy
- Whether to use rich for dry-run table output or plain text formatting

</decisions>
